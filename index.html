Building Enterprise SSO with Stytch in a Rails API: The Complete Guide

How we implemented SAML/OIDC, OAuth, Magic Links, and TOTP MFA in production

May 24, 2025

Single sign-on sounds simple — until you're deep into SAML attribute namespaces at 11pm wondering why Entra ID and Okta send user emails in completely different fields. This article walks through how we built a production multi-tenant SSO system in Rails using Stytch, covering B2B enterprise SSO, B2C OAuth, magic links, and TOTP MFA.

Why Stytch?

We evaluated several authentication platforms before landing on Stytch. The key differentiator: Stytch treats B2B and B2C as genuinely separate products — each with their own project, SDK client, and token namespace. That matters when your platform serves both enterprise clients (who want SAML/OIDC via Okta or Entra ID) and individual users (who just want "Sign in with Google").

The alternative — cramming both into one auth system — creates subtle bugs when org-level MFA policies bleed into consumer flows.

Architecture Overview

The system has three authentication flows:

Flow Who How
B2B SSO Enterprise orgs SAML/OIDC via Stytch B2B
B2C OAuth Individual users Google / Microsoft OAuth
Magic Links New org members Stytch B2B invite emails
TOTP MFA MFA-required orgs Authenticator app (6-digit code)

Each organization in our system has two Stytch identifiers stored on the record: stytch_organization_id and stytch_connection_id — the connection being the specific SAML/OIDC configuration for their IdP. When a user logs in, we look up their organization by email domain and use those two IDs to route them to the right IdP.

Step 1: Two Clients, Not One

The first architectural decision was separating Stytch clients for B2B and B2C into a shared Rails concern. Enterprise SSO uses StytchB2B::Client; individual OAuth uses Stytch::Client pointed at a separate B2C project. Both are lazy-initialized with ||= so they're memoized per-request. 

module StytchClientConcern
  extend ActiveSupport::Concern

  def b2b_stytch_client
    @b2b_stytch_client ||= StytchB2B::Client.new(
      project_id: ENV["STYTCH_PROJECT_ID"],
      secret:     ENV["STYTCH_SECRET"]
    )
  end

  def b2c_stytch_client
    @b2c_stytch_client ||= Stytch::Client.new(
      project_id: ENV["STYTCH_B2C_PROJECT_ID"],
      secret:     ENV["STYTCH_B2C_SECRET"]
    )
  end
end

Keeping the projects separate means org-level MFA policies can never accidentally apply to a consumer OAuth login.

Step 2: The B2B SSO Flow

Authorize

When an enterprise user types their email, we look up their organization by domain and return the Stytch SSO start URL. The frontend redirects the user there — Stytch handles the full SAML/OIDC handshake with the IdP. When authentication completes, Stytch redirects back to our callback with a short-lived code.

Callback — The IdP Detection Problem

This is where things get interesting. Azure Entra ID and Okta both speak SAML but send user attributes in completely different namespaces. We auto-detect which IdP we're talking to by checking for an Entra-specific attribute prefix: 

def detect_idp_type(sso_registration)
  if sso_registration.dig(
    "sso_attributes",
    "http://schemas.microsoft.com/identity/claims/objectidentifier"
  )&.first
    :entra_id
  else
    :okta
  end
end

Entra uses long Microsoft XML schema URIs for every attribute. Okta uses short keys like firstName, email, id. You won't find this documented anywhere — it's something you discover by looking at real SAML responses from both providers.

Once we know the IdP, we extract the user's email, name, and ID from the correct attributes, then find or create the user record and establish a Rails session.

Step 3: B2C OAuth — The Microsoft Email Problem

Google OAuth via Stytch is straightforward. Microsoft is not. Stytch's cached user profile doesn't always have the email on first login. When that happens, we fall back to calling Microsoft's Graph API directly using the access token Stytch provides: 

def extract_microsoft_email(resp)
  user_response = b2c_stytch_client.users.get(user_id: resp["user_id"])
  email = user_response.dig("user", "emails", 0, "email")

  if email.nil?
    access_token = resp.dig("provider_values", "access_token")
    uri = URI("https://graph.microsoft.com/oidc/userinfo")
    req = Net::HTTP::Get.new(uri)
    req["Authorization"] = "Bearer #{access_token}"
    res = Net::HTTP.start(uri.host, 443, use_ssl: true) { |h| h.request(req) }
    email = JSON.parse(res.body)["email"]
  end

  email
end
This fallback took an embarrassing amount of time to find. Save yourself the trouble.

Step 4: Magic Links for Passwordless Invites

When an admin invites a new member to an SSO-enabled org, we use Stytch's B2B Magic Links API instead of a traditional invite email. One API call sends the magic link; when the user clicks it, we validate the token and redirect them into the full SAML flow. The invite and the SSO session end up seamlessly connected — with no password ever set.

Step 5: TOTP MFA

Organizations can require MFA for all members. The key concept here is Stytch's intermediate session token — it represents a user who passed SSO but hasn't completed MFA yet. We return it to the frontend alongside the TOTP QR code on first enrollment. The frontend holds it while the user scans the QR code, then sends it back with the 6-digit code to complete authentication. 

# First login: create TOTP and return QR code + intermediate token
totp_resp = b2b_stytch_client.totps.create(
  organization_id: org.stytch_organization_id,
  member_id:       member_id
)

# Subsequent logins: verify the code
b2b_stytch_client.totps.authenticate(
  organization_id:           org.stytch_organization_id,
  member_id:                 member_id,
  code:                      params[:code],
  intermediate_session_token: params[:intermediate_session_token]
)

The intermediate token cleanly separates "passed SSO" from "completed MFA" without any custom state management on our end.

Migrating Existing Users to MFA

When an org retroactively enables MFA, we don't force password resets. Instead, we use Stytch's passwords.migrate endpoint to register the user's existing bcrypt hash the first time they log in — zero downtime, transparent to the user.

Session Management

One thing worth calling out: Stytch gives you a session_token and session_jwt in every successful auth response, but we don't use them for subsequent API requests. We use Rails' standard cookie-based session. The Stytch tokens are returned to the frontend in case it needs them, but our backend validates the Rails session cookie on every request.

Keep your auth boundary clear — don't mix Stytch token validation with your existing session middleware.

Lessons Learned

  1. Entra ID and Okta have completely different SAML attribute schemas. Plan for IdP detection from day one — you can't assume a uniform attribute shape across providers.
  2. Microsoft Graph is your fallback for missing emails. Stytch's cached profile won't always have the email on first Microsoft OAuth login.
  3. Intermediate session tokens are your MFA handshake. Hold them on the frontend between SSO and MFA steps — never persist them server-side.
  4. Separate your B2B and B2C Stytch projects. The forced SDK separation makes you intentional about which flow each request goes through.
  5. Migrate passwords lazily, not in batch. Use passwords.migrate on first login for MFA-required orgs. No downtime, no user-facing change.
